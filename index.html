React系列(一)JSX与组件的生命周期

JSX

  1. 定义：作为js的扩展，以类html的语法，通过babel编译为React.createElement,获取返回的ReactElement对象，最终生成虚拟DOM。
    1. 其中React.createElement本身只是对参数进行格式化。
    2. 而ReactElement对象则是对DOM的描述。
    3. 渲染时：ReactDOM.render(虚拟DOM,真实DOM)

组件生命周期

  1. 组件特点：’封闭’-渲染；’开放’-组件间通信。
  2. 生命周期：以render为核心
  3. React15:
    1. 挂载(仅发生一次)：constructor;ComponentWillMount;render;ComponentDidMount(异步请求,操作真实DOM)
    2. 更新：
      1. ComponentWillReceiveProps(父组件触发);
      2. ShouldComponentUpdate(自身set state触发);//性能优化
      3. ComponentWillUpdate;render;ComponentDidUpdate
      4. 重点：父组件的重新渲染或props的改变会导致子组件的渲染。
    3. 卸载：ComponentWillUnmount
      • (组件在父组件中被移除；key值变化)
  4. React16.4:
    1. 挂载:
      1. static getDerivedStateFromProps(prop, state)
      2. 取代ComponentWillReceiveProps, 为静态方法（函数式？），实现state的定向更新。
      3. render + ComponentDidMount
    2. 更新:
      1. GetSnapshotBeforeUpdate取代ComponentWillUpdate,生成值会传给ComponentDidUpdate,同时获取更新前的真实DOM及更新后的state&props;
    3. 卸载：不变
  5. 核心变化：适应Fiber架构，将render和commit进行拆分，其中render可以被打断，实现异步渲染。
    1. 重复执行render使得上述一些方法不适合被执行。

组件间通信

  1. props传递:父-子，子-父，兄弟（父-子+子-父）间通信。
    • 不适合多层嵌套的组件关系。
  2. 发布-订阅模式：
    • 手撕代码：
  3. Context API:
    1. 旧API：provider和consumer区分不明显, 容易导致数据不同步
    2. 新API: 数据会穿透ShouldComponentUpdate，实现同步
      • 相关代码：
  4. Redux: Action -> Reducer -> Store -> View，单向数据流。

JS系列(一)执行环境

前篇对浏览器的理解还没有解释好作用域等概念，这一篇希望结合这段时间的学习，对JS代码的执行做一个梳理。

首先回顾一下上一篇里提及的执行环境。代码的执行发生在浏览器的JS解释器/引擎中，涉及到的两大块内存为堆内存(heap memory)和栈内存(call stack)。其中堆内存用来存放引用类型的数据，对应的地址存放在栈中。基本类型的数据存放在栈中。

在执行前，重要的一步是进行预编译，首先创建一个全局上下文，默认会有全局对象window及this变量，指向全局对象。其次，对全局中的变量及函数进行绑定，并且函数中的[scope]属性，也被设置成了对上级作用域的引用，形成了作用域(链)，决定了执行时变量的访问规则。这一阶段实现了所谓的’变量提升’，具体的绑定过程可以回顾上一篇的预编译部分。同时会

接下来，在全局中声明的变量，都会成为全局对象的属性，比如var a = 1;可以通过window.a进行访问。

执行函数时，会创建函数上下文并推入调用栈（不是储存变量的栈？），函数执行完毕时推出调用栈（不一定销毁）。在这个过程中，函数上下文会创建参数对象arguments（注意，这是一个对象，包含length属性）以及this（不再绑定为固定对象）。可以通过[scope]的引用访问作用域链上的变量。如果函数中的嵌套函数引用了不属于当前作用域上的自由变量，则形成了闭包，此时函数的作用域不会销毁。

执行过程中，this的指向一般由调用其方法的对象决定。特殊情况是箭头函数中的this，指向定义时的对象。

  • 三种特殊情景指向window: 立即执行函数，setTimeout, setInterval。
  • 改变this指向：apply,bind,call。

[ES5&6]:

  1. let,const: 块级作用域/花括号；不存在变量提升（不会变成undefined），但是存在暂时性死区(Reference error: cannot access before initialization)。

[常见定义]:

  1. 作用域：储存和访问变量的规则。包含全局、函数、块级作用域。/ 当前所处的执行上下文。
    1. 词法作用域：静态作用域；JS默认的作用域。
    2. 动态作用域。
  2. 闭包：引用了自由变量的函数。
  3. LHS, RHS:查询变量的两种方式。前者为赋值或写入内存，后者为查找或从内容中读取。
  4. 立即执行函数
  5. 调用栈
  6. [ ]详细解释：https://www.html.cn/archives/7262

初见浏览器

这次对浏览器的一个学习，主要目的是为了理解JS代码的执行环境，希望能解决关于作用域和词法环境的问题。

首先了解浏览器的构成。可以通过分析从用户输入url到呈现内容这一行为对浏览器进行拆解。对应此处的分析，可以分为以下几块：

  • 用户界面。例如菜单栏，滚动条等等，接受用户的输入。
  • 浏览器引擎。将用户的输入传递给呈现引擎。
  • 呈现引擎。通过网络获取资源后，进行解析、构建、布局和绘制。
    • html有专门的解析算法（标记化及树构建），而css则可以用常用的解析方法（词法分析，语法分析）。
    • 需要了解对应的性能优化方法。
  • JS解释器。基本组成包含Call stack和heap memory。执行机制为事件循环。
    • 运行三部曲：语法分析，预编译，解析执行。
    • 预编译。
      • 全局预编译：创建global object对象；变量声明；函数声明
      • 函数预编译：创建activity object对象；形参和变量声明；形参和实参统一；函数声明。
      • 决定[作用域]。
  • UI后端。绘制相关。
  • 网络。
  • 数据储存。

从另一个角度(?)来说, 浏览器的线程包括GUI渲染、JS引擎、事件触发、定时器触发、异步http请求线程。其中GUI渲染和JS互斥，这也是导致阻塞的原因（长时间未进行渲染）。
